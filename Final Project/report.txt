Automated Brute-Force Threat Detection and Log Analysis
CPT201 - Final Project Report
Author: Keegan | April 2026
============================================================


1. PURPOSE AND OBJECTIVES
--------------------------

Credential-based attacks - brute-force login attempts, password
spraying, Kerberos pre-authentication abuse, and pass-the-hash
exploitation - consistently rank among the most common initial
access techniques in enterprise threat intelligence reporting.
Manual review of Windows Security event logs is slow, error-prone,
and does not scale across environments with dozens or hundreds of
endpoints. A single overnight brute-force campaign against an
administrator account can generate thousands of event log entries
that an analyst must sift through manually the next morning.

The objective of this project is to build a two-stage automated
pipeline that removes that manual burden. Stage 1 uses PowerShell
to collect authentication events from Windows Security logs - either
locally or via PS Remoting on a remote host - and enriches each
record with a per-account risk score. Stage 2 uses Python to
aggregate the enriched dataset, classify overall incident severity,
and produce both a styled HTML dashboard for analyst review and a
compact JSON summary ready for ingestion into a SIEM or ticketing
system.

Specific goals:
  - Collect Windows Security events (IDs 4625, 4648, 4771, 4776,
    and 4740) over a configurable look-back window.
  - Classify per-account risk as LOW, MEDIUM, or HIGH based on
    failure frequency within the analysis window.
  - Deliver a human-readable HTML report and machine-readable JSON
    output from a single Python command.
  - Demonstrate end-to-end error handling without manual
    intervention, including PS Remoting failures, missing
    dependencies, and empty result sets.


2. DESIGN AND SCRIPTING DETAILS
---------------------------------

The pipeline is divided across two scripts.

threat_detector.ps1 (PowerShell)

The script accepts parameters for target machine, look-back days,
event cap, and a configurable brute-force failure threshold
(default: 5). It detects whether the target is local or remote and
uses PS Remoting for remote queries, falling back to a direct
Get-WinEvent call if remoting is unavailable. This fallback is
critical for environments where WinRM is not universally configured.

After retrieval, each raw WinEvent object is parsed: the username
is extracted from the event message using a regex match on
"Account Name:", placeholder values and machine accounts (those
ending in "$") are discarded, and the event ID is mapped to a
human-readable label. Risk scoring runs as a post-processing
aggregation step: failure counts per user are tallied across
event IDs 4625 and 4771, and each account is labeled HIGH (>= 5
failures), MEDIUM (>= 3), or LOW (fewer). Account lockout events
(4740) are unconditionally labeled HIGH because a successful lockout
confirms an active attack sequence. Explicit-credential logons
(4648) are labeled MEDIUM because they frequently indicate stolen
credential reuse.

A built-in Demo mode injects twelve synthetic events covering all
five monitored Event IDs, allowing the full pipeline to be tested
and demonstrated without access to a production Security log.

analyze_threats.py (Python)

The Python stage reads the CSV export using the standard csv and
pathlib libraries. It then aggregates event counts by type and risk
level, accumulates per-user failure totals, and determines the
overall risk classification for the analysis window. The psutil
library optionally adds a live system snapshot - CPU percentage and
RAM utilization - to help analysts determine whether the analysis
host itself is under load during an active incident.

The script renders a fully self-contained HTML report. Every piece
of user-supplied data is passed through Python's built-in
html.escape() before insertion into the HTML template, preventing
cross-site scripting (XSS) vulnerabilities if the report is ever
served through a web interface. This directly addresses OWASP
Top 10 category A03 (Injection). The script also writes a JSON
summary that mirrors the analysis structure for downstream SIEM
integration. Exit codes 0, 1, and 2 correspond to LOW, HIGH, and
CRITICAL risk respectively, enabling CI/CD pipelines or scheduled
tasks to act programmatically on detected severity.


3. TESTING RESULTS AND TROUBLESHOOTING
----------------------------------------

Testing was conducted in three phases.

Phase 1 - Demo mode validation.
Running threat_detector.ps1 with the -Demo flag generated twelve
synthetic events covering all five monitored Event IDs. The risk
scoring correctly labeled "administrator" (6 failures) as HIGH and
"jdoe" (3 Kerberos pre-authentication failures) as MEDIUM. The
account lockout for "svcbackup" was independently labeled HIGH.
The Python analyzer produced a matching HTML report with an overall
risk level of HIGH and correctly displayed alert banners for
the lockout and explicit-credential events.

Phase 2 - Real log query (local machine).
The script was run against the local Security log with a 7-day
window on a development workstation. No authentication failures
were present in that window, so zero events were returned. The
script exited cleanly with the message "No matching security
events in the specified window," and recommended using -Demo mode,
confirming the zero-results path was handled without crashing.

Phase 3 - Error handling.
Supplying an unreachable hostname to -RemoteComputer triggered the
PSRemotingTransportException catch block. The script printed a
diagnostic message, displayed the Enable-PSRemoting configuration
hint, and fell back to the local log without user intervention.
On the Python side, removing security_events.csv before running
analyze_threats.py confirmed the file-not-found path displayed
clear setup instructions and exited with code 1 rather than raising
an unhandled exception. Uninstalling psutil from the test environment
confirmed the graceful fallback message appeared in the report
without breaking execution.


4. ETHICAL AND COMPLIANCE CONSIDERATIONS
------------------------------------------

Collecting and monitoring authentication logs carries significant
legal and ethical responsibilities. Several frameworks govern how
this pipeline must be deployed.

NIST SP 800-53 (AU-2, AU-6, SI-4): Requires organizations to
define auditable events, review logs for anomalies, and monitor
systems for indicators of attack. This pipeline directly supports
the AU-6 (Audit Record Review) and SI-4 (System Monitoring)
controls by automating detection and surfacing findings to analysts.

GDPR (Articles 5 and 32): Authentication logs inherently contain
personal data because usernames are often tied to natural persons.
Under GDPR, processing must have a lawful basis, must be limited
to a defined purpose, and data must not be retained longer than
necessary. Any organizational deployment of this pipeline must be
accompanied by a documented retention policy, explicit authorization
from data governance leadership, and notification to employees that
authentication activity is monitored.

HIPAA Security Rule (164.312): In healthcare environments, systems
containing protected health information must implement access
controls and maintain audit logs. This pipeline directly supports
the Audit Controls and Access Control implementation specifications
by providing automated detection of unauthorized access attempts.

Principle of Least Privilege: The Security event log requires
"Event Log Readers" group membership or Local Administrator rights.
Credentials used by the script should be stored in Windows
Credential Manager or an enterprise secrets vault and should never
be hardcoded. The script stores no credentials and requests no
network connections beyond the targeted WinRM endpoint, limiting
its attack surface.

Every log query run should be logged, timestamped, and reviewable
by security leadership. The same techniques that enable blue-team
defense can be misused if deployed without authorization requirements.


5. FUTURE APPLICATIONS AND ENHANCEMENTS
-----------------------------------------

Several enhancements would make this pipeline production-ready.

Real-time alerting: Integrating with Microsoft Teams webhooks or
an SMTP relay would push HIGH-risk alerts to an on-call analyst
immediately when the brute-force threshold is crossed, rather than
waiting for a scheduled scan.

SIEM integration: The JSON output is already structured for
machine ingestion. A thin connector to a Splunk HTTP Event
Collector endpoint or an Azure Sentinel workspace would enable
correlation of authentication events with network flow data and
endpoint telemetry for richer threat context.

Baseline deviation detection: Rather than a fixed failure
threshold, a 30-day rolling baseline per account combined with
statistical deviation scoring (Z-score > 2) would reduce false
positives for service accounts with legitimate high-frequency
authentication patterns while flagging genuinely anomalous spikes.

Scheduled execution: Deploying the PowerShell script as a Windows
Scheduled Task running every 15 minutes, with the Python analyzer
running immediately after, would give SOC analysts a continuously
refreshed dashboard at shift start without manual invocation.

Multi-platform support: Extending the Python analyzer to also
accept Linux auditd log files or macOS Unified Log exports would
allow a single reporting framework to cover heterogeneous enterprise
environments from one analyst workstation.
